# Landing Architecture

## Purpose

The landing page is not a marketing surface.

It is a systems statement.

The work leads.
Identity confirms.

The page must feel calm, precise, and controlled.

---

## Core Thesis

The landing page communicates:

- Architectural thinking.
- Deterministic execution.
- Operational discipline.
- Measured technical depth.

It must not communicate hype, personality-first framing, or startup energy.

---

## Structural Order

The landing page must follow this order:

1. Thesis statement (H1).
2. Controlled subline.
3. Minimal CTA row.
4. Divider.
5. Architecture philosophy section.
6. Featured case study.
7. Featured simulation.
8. Terminal block.
9. Identity block (bottom).
10. Footer (version stamp + links).

No hero section.
No oversized imagery.
No autoplay elements.

---

## Visual Hierarchy Rules

- H1 appears once.
- Section headers use H2 only.
- Short paragraphs (1–4 lines).
- Generous whitespace.
- Code blocks allowed.
- Mathematical notation allowed.
- No large walls of text.

The page should read like a clean engineering document.

---

## Tone Constraints

Allowed:

- Declarative statements.
- Technical language.
- Constraint framing.
- Measured claims.
- Clear structure.

Disallowed:

- Playful phrasing.
- Startup hype tone.
- Over-explanation.
- Excessive adjectives.
- Exclamation points.

The voice is controlled and deliberate.

---

## Interaction Discipline

Nothing auto-runs.

- Terminal boots only on visibility or interaction.
- Demos load only on explicit click.
- Only one demo runs at a time.
- Previous demo iframe is destroyed on switch.
- No hidden background computation.

Interaction must demonstrate lifecycle awareness.

---

## Terminal Block Principles

The terminal is a subtle signal.

It must:

- Feel like a real development environment.
- Use manual tab switching.
- Default to a non-vanilla example (attest pipeline).
- Avoid animation excess.

It is not decoration.
It is proof of thought process.

---

## Performance Budget

Landing page constraints:

- Minimal JavaScript.
- No demo bundles included in shell.
- Terminal code footprint kept small.
- Images optimized and lazy where appropriate.
- No blocking external dependencies.

The shell must remain fast regardless of demo complexity.

---

## Provenance Visibility

The landing page must subtly expose:

- Short git SHA.
- Build timestamp (UTC).
- Link to `/meta`.

This is not promotional.
It is verification.

---

## Identity Placement

Identity belongs near the bottom.

Centered.
Minimal.
Headshot small.

Content:

- Name.
- Title.
- Links.

No biography paragraph on landing.

---

## Cognitive Goal

After viewing the page, the visitor should think:

- This person understands systems.
- Nothing here is accidental.
- The architecture is intentional.
- The implementation is disciplined.

The best architects are felt, not seen.
